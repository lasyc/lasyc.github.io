资源操作

Java 的标准 java.net.URL 类以及各种 URL 前缀的标准处理程序，不幸的是，对于所有对底层资源的访问来说，都不是很足够。例如，没有标准化的 URL 实现可用于访问需要从类路径或相对于 ServletContext 获取的资源。虽然可以为专用的 URL 前缀注册新的处理程序（类似于现有 http: 等前缀的处理程序），但这通常非常复杂，并且 URL 接口仍然缺少一些理想的功能，例如检查所指向资源是否存在的方法。

Spring 的 Resource 接口位于 org.springframework.core.io. 包中，旨在成为一个更强大的接口，用于抽象对底层资源的访问。

Resource

Resource 只是一个接口，提供了以下的方法，我们在使用时要使用其实现类

返回类型 方法名 说明
boolean exists() 资源是否以物理形式实际存在
boolean isReadable() 资源是否可读
boolean isOpen() 资源是否关联到一个已打开的流
boolean isFile() 资源是否为文件系统中的文件
URL getURL() 返回资源的 URL 句柄
URI getURI() 返回资源的 URI 句柄
File getFile() 返回资源的 File 句柄

long contentLength() 返回资源的内容长度
long lastModified() 返回资源的最后修改时间戳
Resource createRelative(String) 创建相对于当前资源的资源

ResourceLoader

上边的实例中，我们如果想要获取不同类型的资源，就需要使用对应的实现类，但是 ResourceLoader 解决了这个问题。

ResourceLoader也是一个接口，我们可以通过其显现类的实例对象来调用 getResource 方法获取 Resource 对象，根据传入的 path 前缀获取的 Resource 对象不同，让你能在代码中不硬编码资源读取方式，只写一个路径字符串，框架自动帮你选对实现。

前缀 返回的 Resource 实现
classpath: ClassPathResource
file: FileSystemResource
http: / https: UrlResource
无前缀 取决于 ApplicationContext 类型

注意：ApplicationContext 本身就是 ResourceLoader 这是最容易被忽略的。Spring 的 ApplicationContext 继承了 ResourceLoader，所以你在任何能拿到 ApplicationContext 的地方，都能直接加载资源。

ResourceLoaderAware

上边提到了 ApplicationContext 本身就是 ResourceLoader，所以 spring 项目启动时，Spring 容器 会把自己注入到 ResourceLoader 中，所以我们在 99% 的场景下都可以通过自动注入 ApplicationContext 来获取 ResourceLoader

但是如果我们无法获取 ApplicationContext 怎么办？ResourceLoaderAware 接口提供了一个 void setResourceLoader(ResourceLoader resourceLoader); 方法，在项目启动时，ApplicationContext 会将自己注入到 setResourceLoader 方法中，此时我们可以创建一个类实现 ResourceLoaderAware 接口，实现 setResourceLoader 方法
